# InteractiveClaudeManager sibling: PTY spawn via inherited base machinery + tail->normalize->identical output + turn-end-driven completion

## Objective

Implement the `InteractiveClaudeManager` body — the runnable interactive substrate that is the core engine this epic exists to build. It is a sibling of `ClaudeCodeManager` that extends `AbstractCliManager` (`main/src/services/panels/cli/AbstractCliManager.ts`) and overrides ONLY the abstract hooks, inheriting the still-LIVE PTY machinery — `spawnPtyProcess` (~553), `setupProcessHandlers` (~655), `killProcessTree` (~848) — verbatim. It drives a real subscription-billed interactive `claude` (no `-p`, no `--output-format stream-json`), recovers structured panel fidelity by starting a `TranscriptTailSource` (created in TASK-807) whose already-normalized lines flow `narrow -> router.emitForRun(runId) -> emit('output',{panelId,sessionId,type:'json',data,timestamp})` byte-identical to the SDK manager (`claudeCodeManager.ts:383-389`), and resolves completion against a REPL that NEVER self-exits by reacting to the TranscriptSource `onTurnEnd` signal (write EOF/`/exit` to PTY stdin; the inherited `onExit` + a short transcript-drain settle window is the teardown CONSEQUENCE that resolves the inherited spawn promise so RunExecutor fires `drained -> restAwaitingReview` unchanged). This task is INDEPENDENT of IDEA-029 and startable now (after TASK-807 merges) in parallel with that epic; it touches ONLY the three new files listed in `files_owned` plus the TASK-807 transcript files as read-only collaborators. It does NOT touch `index.ts`, `claudeCodeManager.ts`, `runExecutor.ts`, or `mcpQueryHandler.ts` — the substrate-aware dispatch + facade source (S4/TASK-809) and the shell-hook gating (S5/TASK-810) are separate depends-on-MERGE tasks. This task adds NO duplicate of any IDEA-029 code.

## Implementation Steps

1. **Create `main/src/services/panels/claude/interactiveClaudeManager.ts`** (new). Declare `export class InteractiveClaudeManager extends AbstractCliManager`. Constructor mirrors `ClaudeCodeManager` — `(sessionManager: SessionManager, logger?: Logger, configManager?: ConfigManager, db: Database.Database)` — and stores `db` for the per-run `RawEventsSink`. Reuse the SDK manager's imports as a template: `EventRouter, RawEventsSink, TypedEventNarrowing` from `../../streamParser`, `ApprovalRouter` / `QuestionRouter` from the orchestrator, `getShellPath, findExecutableInPath` from `../../../utils/shellPath`. Import the `TranscriptSource` interface + `TranscriptTailSource` + `encodeCwd` from `./transcript/*` (TASK-807). Do NOT import `@anthropic-ai/claude-agent-sdk` (interactive has no SDK surface).

2. **Implement the abstract hooks:**
   - `getCliToolName()` -> `'Claude Code (Interactive)'`.
   - `testCliAvailability(customPath?)` -> probe the REAL `claude` binary: resolve PATH via `getShellPath()`, then `findExecutableInPath('claude', ...)` honoring `customPath ?? this.configManager?.getConfig()?.claudeExecutablePath`. Return `{ available, error?, version?, path? }`. Unlike `ClaudeCodeManager` (whose SDK substrate is always available), a missing binary MUST surface `{ available: false }` so the inherited `spawnCliProcess` startup path fails loudly.
   - `getCliExecutablePath()` -> the resolved path from the availability probe / config.
   - `buildCommandArgs(options)` -> interactive argv: NO `-p`, NO `--output-format`. Add `--model <m>` ONLY when `options.model && options.model !== 'auto'` (mirror `claudeCodeManager.ts:463`). Add `--strict-mcp-config` when `options.strictMcpConfig` (the base supports the flag). Add `--mcp-config <path>` / `--settings <path>` to inject the cyboflow MCP stdio entry and isolate from user settings (the actual settings/hook file is generated in S5; here emit the flag against the path the writer will own). Document the `settingSources` read-vs-isolate decision in a parity-table comment.
   - `initializeCliEnvironment(options)` -> return env including `CYBOFLOW_RUN_ID` / `CYBOFLOW_ORCH_SOCKET` when `this.orchSocketPath` is set. REUSE the `setOrchSocketPath(socketPath)` seam pattern from `claudeCodeManager.ts:105` (add the same `private orchSocketPath: string | null = null` field + setter). NOTE: S6/TASK-811 asserts `CYBOFLOW_RUN_ID === workflow_runs.id`; this task only wires the env passthrough — it does NOT re-touch `composeMcpServers` (owned by TASK-800).
   - `getCliEnvironment(options)` -> any additional interactive env (default `{}`).
   - `parseCliOutput(...)` -> return `[]`. The interactive substrate does NOT parse raw PTY stdout for structured events (that is the noise terminal stream); structured events come exclusively from the TranscriptSource. The inherited `setupProcessHandlers.onData` (`AbstractCliManager.ts:660`) will call this per-line and get `[]`, which is correct — no panel events from raw PTY.
   - `cleanupCliResources(sessionId)` -> stop the run's `TranscriptSource`, clear `ApprovalRouter`/`QuestionRouter` pending for the runId, and invoke the in-flight-approval-socket deny/close hook (an extensible seam; the socket-deny body lands in S5/TASK-810). Remove the generated settings.json hook entry (the removal call is a seam; writer body in S5).

3. **Add a per-run pipeline + tail-source map** mirroring `ClaudeCodeManager.pipelines` (`claudeCodeManager.ts:120`): `private readonly pipelines = new Map<string, { router: EventRouter; sink: RawEventsSink; runId: string }>()` and `private readonly tailSources = new Map<string, TranscriptSource>()` keyed by panelId. Construct `TypedEventNarrowing` once (PASS the logger — `new TypedEventNarrowing({ verbose: (m) => this.logger?.verbose(m) })` or the project's logger adapter — never `new TypedEventNarrowing()`).

4. **Override `spawnCliProcess(options)`** (override, like `claudeCodeManager.ts:237`). Resolve `runId` from the session row (`sessionRow?.run_id ?? panelId`, the RunExecutor invariant). Build the per-run `router = new EventRouter()` + `sink = new RawEventsSink(this.db, this.logger)`, `sink.attachToRouter(router, runId)`, store in `pipelines`. Call the inherited `await this.spawnPtyProcess(cliCommand, args, worktreePath, env)` to get the `IPty` (do NOT reimplement spawn; do NOT call the base `spawnCliProcess` super since you need the tail wiring interleaved — instead replicate the small availability+args+env preamble from `AbstractCliManager.ts:130-188`, then call `spawnPtyProcess` + `setupProcessHandlers` directly). Start a `TranscriptTailSource` resolving `~/.claude/projects/<encodeCwd(worktreePath)>/` (TASK-807 owns discovery), store it in `tailSources`, and set `onLine = (normalizedLine) => { const typed = this.narrowing.narrow(normalizedLine); router.emitForRun(runId, typed); this.emit('output', { panelId, sessionId, type: 'json', data: normalizedLine, timestamp: new Date() }); }` — field-identical to the SDK envelope. After `waitForFirstLine(DISCOVERY_TIMEOUT_MS)` (or once the PTY is ready), write the initial prompt into PTY stdin via the inherited `sendInput(panelId, prompt + '\n')`. Persist the tail-discovered session UUID to `claude_session_id` via the sessionManager seam, documenting the single-writer-per-substrate rule (interactive writes from the filename; SDK writes from the event at `sessionManager.ts:590` — they never both run for one run).

5. **Completion wiring** (folds the REPL-never-exits blocker). Wire the TranscriptSource `onTurnEnd` callback: on the first turn-end signal, write EOF/`/exit` to PTY stdin (`sendInput(panelId, '')` and/or `'/exit\n'`) to end the REPL turn. Do NOT resolve the spawn promise on `onTurnEnd` directly. Instead, hold a deferred whose `resolve` is called only from the inherited `setupProcessHandlers.onExit` path AFTER a short transcript-drain settle window (`setTimeout(..., SETTLE_MS)`) — so a hung PTY awaiting input (no turn-end) never spuriously resolves. In the `finally`/settle, inspect the PTY exit code: non-zero -> emit the `error`/`exit` shape that drives RunExecutor `failed`; zero -> the inherited `exit` event drives `drained -> restAwaitingReview`. The executor itself is unchanged and never auto-completes.

6. **raw_events ownership.** Because the manager owns the per-run `RawEventsSink` (step 3-4), the RunExecutor bridge for interactive runs MUST run with `skipPersistence: true` (the same contract the SDK manager relies on — `runEventBridge.ts:89,146`). This task does NOT wire the bridge (that is S4/TASK-809), but it MUST guarantee exactly one INSERT per line happens via the manager-owned sink — asserted by the fake-db test. Do not emit a second persistence path.

## Acceptance Criteria notes

- **Inherited-not-overridden:** the grep AC for `spawnPtyProcess|setupProcessHandlers|killProcessTree` must show NO method *declarations* in the new file — `spawnPtyProcess`/`setupProcessHandlers` may appear only as call sites inside `spawnCliProcess`. `killProcessTree` should not appear at all (it is reached via the inherited `onExit`/`killProcess`). These three base methods are LIVE and load-bearing for this sibling (S7/TASK-812 adds the CLAUDE.md note that they must not be pruned).
- **Output envelope must be field-identical to the SDK path** so `runEventBridge` and the structured Claude panel need ZERO edits: `{ panelId, sessionId, type: 'json', data, timestamp }` exactly as `claudeCodeManager.ts:383-389`. The `data` payload is the ALREADY-normalized line from the TASK-807 normalizer — this manager does not normalize; it only narrows + emits.
- **Completion is turn-end-driven, PTY-exit is the consequence (Probe C / Q4):** interactive `claude` writes NO terminal `{type:'result'}` line and the REPL does not self-exit. PTY quiescence is explicitly REJECTED as the primary signal (a hung PTY awaiting input looks finished). The settle window exists only to prevent tail truncation; it is NOT the completion signal.
- **single-writer-per-substrate:** only ONE code path may write `claude_session_id` for a given run. Interactive writes the UUID parsed from the transcript filename; the SDK event-driven write (`sessionManager.ts:590`) belongs to the SDK substrate. Document this and assert the interactive manager does not invoke the SDK event-derived write.
- **logger MUST be passed** to `TranscriptTailSource`, `RawEventsSink`, and `TypedEventNarrowing` (CLAUDE.md observability rule — omitting silently no-ops diagnostics). Tests intentionally exercise the no-logger path, so the grep/audit is for production code only.
- **Per-option parity table** is a required comment block enumerating model / permissionMode / strictMcpConfig / settingSources / resume / systemPromptAppend decisions. `systemPromptAppend` has NO interactive append channel and is delivered via prompt-body prepend in S6/TASK-811 — this task only notes that and does not implement the prepend.
