# Interactive permission gating via a net-new async-deferred socket handler + .claude/settings.json shell hook

## Objective

Preserve cyboflow's review-queue differentiator on the interactive substrate — the only substrate that survives the 2026-06-15 Agent-SDK-credit split — by routing interactive `claude` tool calls through the SAME `ApprovalRouter` the SDK path uses. The mechanism (PRIMARY, gated on S0 Probe A PASS) is a `.claude/settings.json` PreToolUse `*` shell hook (`preToolUseShellHook.ts`, a standalone ASAR-unpacked node script resolved via the `scriptPath.ts` pattern) that posts `{type:'shell-approval-request',...}` over `CYBOFLOW_ORCH_SOCKET` and blocks for the FULL human-decision window via a heartbeat, plus a NET-NEW async-deferred branch in `mcpQueryHandler.ts` — the first handler in the system that does NOT `writeResponse` synchronously: it applies `isToolAllowed(loadMergedPermissionRules(worktree))` first (short-circuit allow with no approval row, SDK parity), then awaits `ApprovalRouter.requestApproval` with a `socketReply` that writes the verdict back on the held-open socket. AskUserQuestion has no shell-hook `updatedInput` channel, so it is native-TUI-only and explicitly NOT routed through QuestionRouter on this substrate. If Probe A FAILED, the FALLBACK body ships no roll-our-own gating (native TUI behind a banner; SDK substrate keeps full gating) and a test asserts no non-functional hook is written.

This is a depends-on-MERGE task. `mcpQueryHandler.ts` is OWNED by IDEA-029 TASK-802 (which adds `cyboflow_report_step`): the new `shell-approval-request` branch lands as part of / AFTER TASK-802 merges — this task branches off the merged TASK-802 tree and NEVER co-edits `mcpQueryHandler.ts` concurrently, and adds NO duplicate of any IDEA-029 code. The transport (`OrchSocketServer`, TASK-798) and `setOrchSocketPath` boot wiring (TASK-799) are CONSUMED, not re-implemented — this task adds NO new socket server. `CYBOFLOW_RUN_ID=workflow_runs.id` (TASK-800) is a precondition (without it, every approval binds a non-existent row -> fail-closed deny-everything); a precondition assertion guards that exact silent-failure mode. The interactive manager (`interactiveClaudeManager.ts`, TASK-808/S3) and the substrate dispatch facade (TASK-809/S4) are consumed: this task EXPOSES the per-runId socket-tracking + deny/close cancel affordance that TASK-808's `cleanupCliResources` calls, and writes/removes the hook from `initializeCliEnvironment`/`cleanupCliResources` seams owned by TASK-808 — those manager-side call sites land in TASK-808, not here.

- **Wire protocol.** The hook subprocess sends `JSON.stringify({type,requestId,runId,toolName,toolInput}) + '\n'` and the handler replies with `JSON.stringify({type:'mcp-query-response',requestId,ok:true,data:{permissionDecision:...}}) + '\n'` via `writeResponse` (which already appends `'\n'`, `mcpQueryHandler.ts:228-230`). Do not double-encode. The `requestId` round-trips so the hook can correlate the response on a shared socket.
- **Async-deferred is the novel part.** Every existing `McpQueryHandler` branch calls `writeResponse` synchronously inside the switch. The `shell-approval-request` branch must NOT — it returns after kicking off `requestApproval`, and only the `socketReply` closure writes the response, possibly minutes later. The AC that "TASK-798's fire-and-forget connection lifecycle tolerates a branch that never responds synchronously" is the load-bearing integration check: if `orchSocketServer.ts`'s `'close'`/`'error'` cleanup evicts or destroys the socket on idle, the held-open reply path breaks — assert the socket survives.
- **isToolAllowed short-circuit = SDK parity, no double-prompt.** This mirrors `claudeCodeManager.ts:576-583` exactly: an allow-listed tool returns `allow` with no `ApprovalRouter` round-trip and no approvals row. The matcher (`permissionRules.ts:217-231`) is conservative — non-matches fall through to the router, never the reverse.
- **Fail-closed semantics.** Deny on socket disconnect / orchestrator-down ONLY — distinguished by socket liveness (`'close'`/`'error'`), never a timer. Human slowness must never produce a deny; approvals do not auto-expire (`approvalRouter.ts:29-31`). The fake-timer test proves a multi-minute idle still yields the real verdict.
- **CYBOFLOW_RUN_ID precondition (TASK-800).** If `CYBOFLOW_RUN_ID` is still the Claude session UUID, `requestApproval`'s guarded `UPDATE workflow_runs ... WHERE id=? AND status='running'` (`approvalRouter.ts:219-228`) finds `changes===0` -> throws `RunNotRunningError` -> the hook fails closed (deny-everything). The precondition assertion must surface this loudly rather than silently denying every tool call.
- **Cancel contract.** `clearPendingForRun` (`approvalRouter.ts:378-414`) explicitly does NOT invoke `socketReply` ("the run is being torn down; the socket is no longer meaningful") — correct for the SDK in-process transport but WRONG for the shell transport, where a real socket is blocking a real PTY. Hence the separate deny-and-close affordance (step 5).
- **AskUserQuestion.** A shell PreToolUse hook can only return `permissionDecision` allow|deny|ask; there is NO `updatedInput` channel to inject the user's selected answers the way the in-process SDK `HookJSONOutput` does (`claudeCodeManager.ts:598-619`). So AskUserQuestion is native-TUI-only here — no QuestionRouter pending row, no `awaiting_input` leak.
