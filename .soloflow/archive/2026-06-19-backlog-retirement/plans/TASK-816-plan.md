# Interactive chrome (INTERACTIVE pill, LIVE PTY bar) + first-interaction warn dialog + reduced-motion

## Objective

Add the design chrome and the deliberate guardrail modal for the interactive substrate, per the IDEA-030 hi-fi handoff (`/tmp/cyboflow-interactive-design/design_handoff_interactive_chat/README.md`). This task OWNS one new presentational component â€” `InteractiveWarnDialog.tsx` â€” and adds small ADDITIVE edits to `InteractiveTerminalView.tsx` (created by TASK-815, committed first) for the warn-trigger wiring + chrome. The deliverables are: (1) a first-interaction warning modal, built on the existing `ui/Modal` (NOT a hand-rolled scrim), with a hazard-stripe eyebrow, the warning title/body copy, and two actions â€” primary "Use chat instead" (focuses the composer) and ghost "Interact anyway" (grants terminal focus + flips the per-run keystroke-relay flag) â€” opened on the first mousedown anywhere on the terminal surface and shown at most ONCE per run; (2) the INTERACTIVE pill (terracotta, pulsing dot) on the pane-head and the LIVE PTY session bar (`claude --resume <id>`, pid, tty, elapsed counter, token meter) as presentational chrome around `InteractiveTerminalView`; (3) a `prefers-reduced-motion` guard that drops the pulse/blink/spinner loops. This is pure renderer chrome â€” it makes ZERO production change to the structured Workflow panel or the SDK render path (Q3 panel-preservation), introduces NO `any`, and declares NO tRPC subscription of its own (the per-run `has-warned` + `relay-enabled` flags live in component/store state and are CONSUMED by TASK-817). It does NOT touch any backend file, the raw-PTY pipeline (TASK-814), the xterm mount / `subscribeToPtyBytes` wiring (TASK-815), the live-input relay (TASK-817), the persistence/completion rework (TASK-818), or the approval-gate wiring (TASK-819).

## Acceptance Criteria notes

- **The warn modal MUST use `ui/Modal`, not a hand-rolled scrim.** `Modal` already gives the exact handoff contract ("scrim click dismisses, card click does not") through its mousedown-target guard (`Modal.tsx:78-100`) and the `closeOnOverlayClick`/`closeOnEscape`/focus-management/body-scroll-lock behaviors â€” re-implementing a scrim would duplicate that and risk drift. `InteractiveWarnDialog` is a thin presentational wrapper: Modal + hazard-stripe eyebrow + copy + two `ui/Button` actions.
- **Show-once-per-run is the load-bearing semantic.** The handoff says "Consider only warning on the first direct interaction per session" â€” IDEA-030 locks this as per-run: the first mousedown opens the modal, and the `has-warned` flag suppresses every subsequent open for that run. "Interact anyway" sets `has-warned` AND flips the keystroke-relay flag that TASK-817 reads to start relaying `xterm.onData` verbatim; "Use chat instead" focuses the composer and leaves relay disabled. This task SETS those flags but does NOT consume them for relay (TASK-817 does).
- **Chrome is interactive-only and presentational â€” Q3 panel preservation holds.** The INTERACTIVE pill and LIVE PTY bar render only when `substrate === 'interactive'` and are absent for `sdk`. No production line changes on `WorkflowProgressTimeline.tsx`, `useWorkflowPhaseState.ts`, or `RunRightRail.tsx`; the structured Workflow panel and SDK render path stay byte-identical (assert via `git diff main --stat`).
- **Reduced-motion is an accessibility contract, not cosmetic.** When `prefers-reduced-motion: reduce` matches, the pulsing dots / spinner / cursor blink loops are dropped by NOT applying the animated class. The elapsed counter is information (a numeric value), so it may keep updating â€” only the decorative motion loops are gated.
- **Additive-only edits to shared files.** `InteractiveTerminalView.tsx` and its test are owned by TASK-815 (committed first in this single-worktree sequential chain); this task appends the warn-trigger + chrome + reduced-motion guard and ADD-only test cases. Do NOT remove or reorder TASK-815's xterm mount / `subscribeToPtyBytes` / `term.write` wiring, and do NOT pre-empt TASK-817's `xterm.onData` relay or resize wiring.
- **No tRPC subscription is declared here.** `InteractiveWarnDialog` takes plain callback props; the substrate flag and session metadata are read from props/store that TASK-813/TASK-815 already surface via AppRouter-inferred types. No local mirror interface, no `(evt: unknown)` guard, no untyped `IPCResponse` â€” those CLAUDE.md rules are satisfied by construction since this task introduces no new wire type.
