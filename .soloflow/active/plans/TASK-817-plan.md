# sendTurn live-input: backend relayInput + runs mutation + composer relay + Interact-anyway keystroke relay + PTY resize

## Objective

Add the ONLY post-spawn input path into a LIVE interactive run, so the xterm.js terminal (TASK-815) is truly two-way and the composer can drive the persistent REPL. Today there is NO sendTurn for runs: `RunExecutor` writes the initial prompt exactly once at spawn (`InteractiveClaudeManager.spawnCliProcess` → `sendInput` at interactiveClaudeManager.ts:509), and the only "continue" paths (`continuePanel`/`restartPanelWithHistory`, interactiveClaudeManager.ts:752-780) KILL and re-spawn a fresh REPL — they do NOT write a follow-up turn into the live PTY. This task introduces a true live-input relay: `SubstrateDispatchFacade.relayInput`/`relayResize` → `interactiveManager.sendInput` / node-pty `resize` into the SAME running process (NOT continuePanel), surfaced as `runs.relayInput`/`runs.relayResize` tRPC mutations, consumed by (a) the ChatInput composer's new ENABLED `workflow-interactive` mode (relays `text + '\n'` as a real REPL turn) and (b) `InteractiveTerminalView`'s `xterm.onData` raw-keystroke relay (VERBATIM, no newline), gated behind TASK-816's first-interaction "Interact anyway" flag, plus a `ResizeObserver`-driven resize relay.

This slice is ADDITIVE on the four shared backend files (per the IDEA-030 execution model — TASK-816 has already committed before this runs): the facade fan-in/dispatch/abort/dispose (substrateDispatchFacade.ts:74-160) is untouched; the runs router stub pattern and existing dep-bags (runs.ts:43-55) are extended, not rewritten; and TASK-814's `ptyPublisher` wiring + the cyboflow:pty channel are preserved in index.ts. The SDK substrate is a strict NO-OP for the relay methods (it has no PTY — `claudeCodeManager.ts` stores `process: undefined as never`), which keeps the structured Workflow panel + SDK path byte-identical (Q3). The persistence/completion rework (gating the turn-end kill, the event-driven rest, the explicit-termination resolver) is DEFERRED to TASK-818 — this task only adds the input transport; it does NOT change when a run completes.

## Acceptance Criteria notes

- **The relay is into the LIVE PTY, never a kill+respawn.** `relayInput` MUST route to `interactiveManager.sendInput` (live `cliProcess.process.write`), NOT `continuePanel`/`restartPanelWithHistory` (those are quick-session fresh-session kill+respawn, interactiveClaudeManager.ts:752-780). Using continuePanel would destroy the persistent session this epic is building.
- **SDK no-op is the Q3 guarantee.** The facade resolves SDK runs to the SDK manager, which has no PTY (`process: undefined as never`). `relayInput`/`relayResize` MUST early-return for SDK so the structured panel + SDK iterator path stay byte-identical. The facade test's SDK-no-op case is what proves this.
- **`runId === panelId === sessionId` is the orchestrator invariant** (substrateDispatchFacade.ts:21,101-102), so `runs.relayInput({runId})` maps directly to `facade.relayInput(panelId = runId, text)` with no lookup. Document this in the mutation comment.
- **The composer `'\n'` vs raw-verbatim split is load-bearing.** The composer relay appends `'\n'` (a complete REPL turn the user submitted via the Send button/Enter). The "Interact anyway" raw-keystroke relay passes bytes VERBATIM (xterm already encodes Enter as `\r`); appending `'\n'` there would double-submit and corrupt the REPL. The tests assert both.
- **The keystroke relay is gated on TASK-816's per-run flag.** Default (flag unset) = `xterm.onData` is inert and the composer is the only input path — this preserves the first-interaction warn-modal guardrail. The flag flips only after the user chooses "Interact anyway".
- **Persistence is NOT this task.** Relaying input into the live REPL is necessary but the REPL still dies at the first turn-end until TASK-818 gates `handleTurnEnd`'s EOF+'/exit' kill. This task does NOT touch `runExecutor.ts`, the turn-end path, or the completion contract — it only adds the input transport. The COMPLETION-MODEL CONTRACT (gate the turn-end kill behind the persistent flag; emit a turn-end EVENT and keep the REPL alive re-armable per-turn; decouple "turn done" from spawn-promise resolution and route the turn-end event through the facade EventEmitter to a new RunExecutor handler that calls restAwaitingReview WITHOUT resolving the spawn promise; make EXPLICIT termination — End-session / Merge / Dismiss / stopPanel — the only spawn-promise resolver via the now-conditional EOF+'/exit' path; keep the SDK path UNTOUCHED; address recoverStaleAwaitingReview killing a live awaiting-input run on boot and teardownRun not disposing the bridge while the REPL is alive; never reuse the turn-end marker as the drained proxy) is delivered VERBATIM by TASK-818 and is out of scope here.
- **Additive on shared files.** TASK-814 (facade fan-in + ptyPublisher) and TASK-816 (ChatInput substrate-thread groundwork / warn flag) have already committed in this same worktree. Do NOT remove or rewrite their wiring — add the relay methods/mutations/dep-bag/composer-mode/onData-relay alongside.
- **AppRouter inference + no `any`.** All tRPC mutation inputs are inferred from the router; no `RelayInput`/`RelayResize` mirror interface in the renderer, no `(evt: unknown)` guard, and the no-`any` grep over `files_owned` must return 0.
